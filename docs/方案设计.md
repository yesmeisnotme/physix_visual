## 1. 项目概述

### 1.1 目标

构建一个程序，读取游戏地图导出的 PhysX 碰撞数据文件（`collision.bin`），在三维场景中进行可视化展示，并支持视角转移（旋转、平移、缩放，可选 FPS 漫游）。

### 1.2 非目标（当前阶段）

- 不进行物理仿真
- 不修改或写回 `collision.bin`
- 不替代游戏内碰撞调试工具的全部功能

### 1.3 交付决策（已定稿 ✅）

采用 **C++ Converter + Three.js Web Viewer** 路线，分四期交付：

| 期 | 交付物 | 用户可见能力 |
|----|--------|--------------|
| **MVP（Phase 0–2）** | `physix_convert.exe` + 样本 `collision.gltf` | 命令行转换 bin；可用外部 glTF 工具或 Phase 3 查看 |
| **可用版（Phase 3）** | 上述 + Web Viewer | 浏览器中 orbit 查看碰撞、统计面板 |
| **完整版（Phase 4）** | 上述 + `--open`、过滤、高亮 | 一条命令转换并打开浏览器；完整调试 UI |

**不做（当前范围）**：原生 OpenGL 一体化 exe、Electron 壳、物理仿真。

### 1.4 里程碑与成功标准

| 里程碑 | 验收概要 | 参考 |
|--------|----------|------|
| **M1** SDK 就绪 | PhysX 3.4 编译通过，Loader 能读样本 bin | [Phase 0–1](#phase-0格式验证--sdk-就绪) |

## 2. 背景与约束

### 2.1 游戏侧加载方式（已确认）

游戏后台使用 PhysX 原生二进制反序列化加载碰撞数据：

```cpp
physx::PxCollection *create_collection_from_bin(const char *data, u32 data_length)
{
    xy_assert_retval(data, NULL);
    PX_UNUSED(data_length);
    return PxSerialization::createCollectionFromBinary((void *)data, *gSerializationRegistry);
}
```

要点：

- `data` 为文件完整内容的内存指针
- `gSerializationRegistry` 由 `PxSerialization::createSerializationRegistry(*physics)` 创建
- 可视化工具必须使用 **与游戏相同大版本的 PhysX SDK（3.4）** 才能可靠加载

### 2.3 游戏 PhysX 版本（已确认）

| 项 | 值 |
|----|-----|
| 游戏使用 | **PhysX 3.4** |
| 本项目 SDK | **PhysX 3.4.2-bsd**（GitHub 官方源码，已下载至 `third_party/PhysX-3.4`） |
| 样本导出版本 | **PhysX 3.4.0**（见 [§3.2](#32-文件头结构官方格式) 头字段 `PX_PHYSICS_VERSION = 0x03040000`） |

> **版本兼容性**：3.4.2 SDK 通常可加载 3.4.0 导出的 bin（`PX_BINARY_SERIAL_VERSION` 均为 0）。若加载失败，需与游戏对齐到完全相同的小版本或 build。

### 2.4 游戏世界约定（已确认）

| 项 | 约定 | 对可视化的影响 |
|----|------|----------------|
| **地图形态** | 整张地图 **一个 bin**，不分 chunk | 无需分块加载/合并；单文件整图加载即可 |
| **坐标原点** | **地图中心**为原点 | 相机初始 target 设为 `(0,0,0)` 或全场景 AABB 中心；网格地面以原点为中心 |
| **长度单位** | **厘米（cm）**，1 PhysX unit = 1 cm | UI 标注单位用 cm；网格间距建议 100 cm（1 m）或 1000 cm 一档；相机远裁剪面按地图尺寸（cm 级）估算 |
| **碰撞类型（游戏已知）** | 主要为 **HeightField + ConvexMesh** | 优先验证、优化这两类；**实现仍覆盖全部 `PxGeometryType`**（Box/Sphere/Capsule/TriangleMesh 等），以备其他地图或遗漏类型 |

### 2.2 核心约束

| 约束 | 说明 |
|------|------|
| SDK 版本绑定 | 二进制格式与 `PX_PHYSICS_VERSION` + `PX_BINARY_SERIAL_VERSION` 绑定，不可手写逆向 |
| 平台绑定 | 样本为 `W_64`（Windows x64），工具首版对齐该平台 |
| 内存对齐 | 反序列化 buffer 须 **128 字节对齐**（`PX_SERIAL_FILE_ALIGN`） |
| 内存生命周期 | 反序列化对象 in-place 引用 buffer，在 `collection` 及对象 `release` 前 **不可释放** buffer |
| 不可纯前端解析 | 浏览器/脚本无法直接解析 SEBD，必须经过 C++ + PhysX 或预转换中间格式 |

## 3. 输入数据格式

### 3.1 格式定性

`collision.bin` 是 **PhysX 原生二进制序列化**（非自研简化格式、非 cooked mesh 流）。

- 写入 API：`PxSerialization::serializeCollectionToBinary`
- 读取 API：`PxSerialization::createCollectionFromBinary`
- 文件 Magic：**`SEBD`**（Serialized Engine Binary Data）

### 3.2 文件头结构（官方格式）

依据 PhysX 3.4 源码 `SnBinarySerialization.cpp` 中的格式说明：

```
偏移        大小    字段
────────────────────────────────────────────
0x00        4       Magic "SEBD"
0x04        4       PX_PHYSICS_VERSION（样本：0x03040000 → 3.4.0）
0x08        4       PX_BINARY_SERIAL_VERSION（样本：0）
0x0C        4       PX_BUILD_NUMBER（样本：0）
0x10        4       platform tag（样本："W_64" = win64）
0x14        4       markedPadding（样本：0）
...         ...     manifest table（对象目录）
...         ...     import / export / internal references
...         ...     object data blobs
...         ...     extra data
```

`PX_PHYSICS_VERSION` 编码（来自 `PxPhysicsVersion.h`）：

```
PX_PHYSICS_VERSION = (MAJOR<<24) + (MINOR<<16) + (BUGFIX<<8) + 0
```

样本 `collision.bin` 头 4 字节 `00 00 04 03` 按小端读为 **`0x03040000`**，即 **3.4.0**（此前误读为 4.3.x，已纠正）。

manifest table 每条记录：`(PxU32 offset, PxU32 type)`，其中 `type` 为 `PxConcreteType`。

### 3.3 加载流程（必须与官方一致）

```cpp
// 1. 初始化 PhysX
PxDefaultAllocator allocator;
PxDefaultErrorCallback errorCallback;
PxFoundation* foundation = PxCreateFoundation(PX_PHYSICS_VERSION, allocator, errorCallback);
PxPhysics* physics = PxCreatePhysics(PX_PHYSICS_VERSION, *foundation, PxTolerancesScale());
PxSerializationRegistry* registry = PxSerialization::createSerializationRegistry(*physics);

// 2. 读取文件到 128 字节对齐 buffer
FILE* fp = fopen("collision.bin", "rb");
fseek(fp, 0, SEEK_END);
unsigned fileSize = ftell(fp);
fseek(fp, 0, SEEK_SET);

void* memory = malloc(fileSize + PX_SERIAL_FILE_ALIGN);
void* memory128 = (void*)((size_t(memory) + PX_SERIAL_FILE_ALIGN) & ~(PX_SERIAL_FILE_ALIGN - 1));
fread(memory128, 1, fileSize, fp);
fclose(fp);

// 3. 反序列化
PxCollection* collection = PxSerialization::createCollectionFromBinary(memory128, *registry);

// 4. 使用完毕后
collection->release();
registry->release();
// 此后才可 free(memory)
```

> **注意**：若游戏侧 `read` 出来的 buffer 已保证 128 对齐，可直接使用；可视化工具侧应显式做对齐分配，不依赖假设。

## 4. 样本分析（collision.bin）

### 4.1 基本信息

| 属性 | 值 |
|------|-----|
| 路径 | 项目根目录 `collision.bin` |
| 大小 | 671,588 字节（约 656 KB） |
| Magic | `SEBD` |
| 导出 SDK 版本 | **PhysX 3.4.0**（`PX_PHYSICS_VERSION = 0x03040000`） |
| 二进制格式版本 | `PX_BINARY_SERIAL_VERSION = 0` |
| 平台 | `W_64`（Windows x64） |
| 对象数量 | manifest 解析 **1837** 条（脚本统计；头字段附近 u32=1838，差 1 待 C++ 加载验证） |

### 4.2 对象类型统计

从 SEBD 对象目录解析（`type` → 推测 `PxConcreteType`）：

| Type | 推测类型 | 数量 | 备注 |
|------|----------|------|------|
| 1 | HeightField | 1 | **游戏已知类型**，地形碰撞 |
| 2 | ConvexMesh | 189 | **游戏已知类型**，本样本主要几何 |
| 7 | Shape | 423 | 碰撞形状 |
| 8 | Material | 1223 | 物理材质（多为默认材质） |
| 9 | 特殊对象 | 1 | 待 PhysX 加载后确认具体类型 |

**未出现的类型（本样本，实现仍须支持）**：

- TriangleMesh（type 3/4）— 本样本无，其他地图可能有
- RigidStatic / RigidDynamic（type 5/6）
- Box / Sphere / Capsule 等基本体 — 可能嵌在 Shape 中，需 C++ 加载后确认

### 4.3 样本特征结论

1. **与游戏认知一致**：HeightField + ConvexMesh；本样本无 TriangleMesh
2. **可能未序列化 RigidActor**，Shape 可能作为独立对象存在于 Collection 中；遍历不能只依赖 Actor→Shape，必须 **直接扫描 Collection 中的 PxShape**
3. **Material 数量远多于 Shape**，属 PhysX 序列化常见现象，可视化时可忽略或合并
4. **无对象名称字符串**，导出时 likely 未启用 `exportNames=true`
5. **整图单文件**：656 KB，423 Shape / 189 ConvexMesh，一次加载即可（**不分 chunk**）

## 5. 总体架构

### 5.1 定稿架构：Converter + Web Viewer

```
collision.bin
    │
    ▼
physix_convert.exe          ← C++，链接 PhysX 3.4.2-bsd
    │  createCollectionFromBinary
    │  遍历 Shape → 提取 mesh（全类型，优先 HF/Convex）
    │  坐标：cm，原点在地图中心
    ▼
collision.gltf (+ 可选 .bin)
    │
    ▼
viewer/（Three.js + Vite）   ← 浏览器 OrbitControls + 调试面板
    │
    ▼（Phase 4）
physix_convert --open       ← 转换完成后自动打开默认浏览器
```

**日常工作流**：

```bash
# 开发期
physix_convert.exe -i collision.bin -o out/collision.gltf
cd viewer && npm run dev          # 拖放或加载 out/collision.gltf

# 工程化后
physix_convert.exe -i collision.bin -o out/collision.gltf --open
```

### 5.2 方案对比

| 方案 | 描述 | 优点 | 缺点 | 建议 |
|------|------|------|------|------|
| **A. C++ 转换 + Web Viewer** | PhysX 加载 → glTF → Three.js | 解耦、易分享、UI 丰富 | 两步流程 | **✅ 已选定** |
| **B. C++ 一体化** | PhysX + OpenGL/ImGui 直接渲染 | 无中间格式 | UI 开发成本高 | 不在当前范围 |
| **C. 纯 Web 解析 bin** | 不链接 PhysX | 无 | 不可行 | **否决** |
| **D. Python 原型** | PyOpenGL 快速验证 | 迭代快 | 性能与维护性一般 | 仅用于早期验证 |

## 7. 数据模型

解析层与渲染层之间的统一中间表示（语言无关）：

```typescript
interface CollisionScene {
  bounds: AABB;                    // 全场景包围盒
  metadata: {
    sourceFile: string;
    physxVersion: string;
    platform: string;
    objectCount: number;
  };
  actors: CollisionActor[];
}

interface CollisionActor {
  id: string;                      // 索引或 name（若有）
  transform: Matrix4;              // 4×4 列主序
  shapes: CollisionShape[];
}

interface CollisionShape {
  id: string;
  type: ShapeType;
  localTransform: Matrix4;
  flags: {
    isTrigger: boolean;
    isSimulation: boolean;
  };
  geometry: GeometryData;
}

type ShapeType =
  | 'box' | 'sphere' | 'capsule' | 'plane'
  | 'convex_mesh' | 'triangle_mesh' | 'heightfield';

interface MeshGeometry {
  vertices: Float32Array;          // [x,y,z, x,y,z, ...]
  indices: Uint32Array;            // 三角形索引
}

interface AABB {
  min: [number, number, number];
  max: [number, number, number];
}
```

**变换合成规则**：

```
worldMatrix = actorTransform × shapeLocalTransform
worldVertex = worldMatrix × localVertex
```

## 8. 几何提取策略

### 8.1 遍历策略（双路径）

```cpp
// 路径 A：按 RigidActor 遍历（常规场景）
for (PxU32 i = 0; i < collection->getNbObjects(); ++i) {
    PxBase* obj = collection->getObject(i);
    if (!obj->is<PxRigidActor>()) continue;
    PxRigidActor* actor = obj->is<PxRigidActor>();
    PxTransform globalPose = actor->getGlobalPose();
    // actor->getShapes(...) → 提取每个 shape
}

// 路径 B：直接扫描 PxShape（本样本可能需要）
for (PxU32 i = 0; i < collection->getNbObjects(); ++i) {
    PxBase* obj = collection->getObject(i);
    if (!obj->is<PxShape>()) continue;
    PxShape* shape = obj->is<PxShape>();
    PxGeometryHolder geom = shape->getGeometry();
    PxTransform pose = shape->getLocalPose();  // 无 Actor 时 pose 即世界变换
    // 按 geom.getType() 分支
}
```

> 参考实现：PhysX SDK 示例 `PhysXSample.cpp` 中的 `createRenderObjectFromShape()`。

### 8.2 各 Geometry 类型处理

**策略**：游戏侧已知 **HeightField + ConvexMesh** 为主；Converter **实现全部类型**，按 `PxGeometryType` 分支，未知地图也能正确显示。

| PxGeometryType | 提取方式 | 优先级 |
|----------------|----------|--------|
| `eHEIGHTFIELD` | samples 网格 → 三角带 | **高**（游戏已知） |
| `eCONVEXMESH` | `getPolygonData()` 扇形三角化 | **高**（游戏已知） |
| `eTRIANGLEMESH` | `getVertices()` + `getTriangles()` | 中（本样本无，须支持） |
| `eBOX` | `halfExtents` × 12 三角 | 低 |
| `eSPHERE` | `radius` → UV sphere | 低 |
| `eCAPSULE` | `radius + halfHeight`（PhysX 默认轴 X） | 低 |
| `ePLANE` | 大 quad 可视化 | 低 |

### 8.3 ConvexMesh 提取示例

```cpp
if (geom.getType() == PxGeometryType::eCONVEXMESH) {
    const PxConvexMeshGeometry& cg = geom.convexMesh();
    PxConvexMesh* mesh = cg.convexMesh;
    const PxVec3 scale = cg.scale.scale;

    PxU32 nbVerts = mesh->getNbVertices();
    const PxVec3* verts = mesh->getVertices();

    PxU32 nbPolys = mesh->getNbPolygons();
    for (PxU32 i = 0; i < nbPolys; ++i) {
        PxHullPolygon poly;
        mesh->getPolygonData(i, poly);
        // poly.mIndexBase, poly.mNbVerts → 扇形三角化
    }
    // 顶点乘以 scale，再乘 shapePose 到世界空间
}
```

### 8.4 坐标系与单位

| 系统 | 约定 |
|------|------|
| PhysX | 左手系，Y 向上 |
| 游戏世界 | **原点在地图中心**；**1 unit = 1 cm** |
| Three.js | 右手系，Y 向上 |
| 导出 glTF | 顶点坐标保持 **cm** 不变（与游戏一致）；UI 显示「cm」；可选提供「显示为 m」切换（÷100） |
| 手性转换 | 导出时若出现镜像，对 Z 取反或按 Sample 惯例调整；用样本整图 AABB 中心是否在原点附近验证 |

**相机初始参数建议**（cm 单位）：

- `target = (0, 0, 0)`（地图中心）
- 根据全场景 AABB 对角线长度 `d`（cm），初始相机距离 ≈ `d × 1.2`
- 网格：`GridHelper(size, divisions)` 如 size=100000（1 km）、divisions=100 → 每格 10 m

| ConvexMesh | 半透明蓝 + 可选线框 |

| Box / Sphere / Capsule | 半透明黄 |

## 9. 可视化与视角交互

### 9.1 MVP 视角能力（Orbit 相机）

| 操作 | 行为 |
|------|------|
| 左键拖拽 | 绕目标点旋转（yaw / pitch） |

## 12. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| SDK 小版本不匹配 | 加载失败或崩溃 | 使用 PhysX 3.4.x；优先 3.4.2-bsd；与游戏 build 对齐 |
| buffer 未 128 对齐 | 反序列化失败 | Converter 显式 aligned alloc |
| buffer 过早释放 | 野指针崩溃 | 严格遵循 release 顺序 |
| 无 RigidActor 仅有 Shape | 漏提取碰撞体 | 双路径遍历（Actor + 直接 Shape） |
| 坐标系/单位不一致 | 尺度或位置偏差 | 1 unit=1 cm、原点在中心；用 AABB 中心≈原点验证 |
| 无对象 name | 调试困难 | 建议游戏侧 `exportNames=true`（可选） |
| Trigger 干扰观察 | 视觉混乱 | 解析 `PxShapeFlag`，默认隐藏 trigger |
| 未来出现 TriangleMesh 大地图 | 性能下降 | 线框默认、视锥剔除（仍单文件加载） |

### 14.3 交付形态对比（存档）

<details>
<summary>Web vs exe 对比（已决策，点击展开）</summary>

#### 架构前提

无论选哪种，**PhysX 加载几乎都必须走 C++**（`createCollectionFromBinary`）。差异在于 C++ 之后怎么走：

```
方案 A（Web）：  collision.bin → [C++ exe 转换] → .gltf → 浏览器 Three.js 查看  ← 已选
方案 B（exe）：   collision.bin → [C++ 一体化] → OpenGL/ImGui 直接渲染
方案 C（混合）：  C++ exe 转换 + --open 自动打开浏览器  ← Phase 4
```

完整对比表见 git 历史 v0.4；最终采用 **A + C**。

</details>
